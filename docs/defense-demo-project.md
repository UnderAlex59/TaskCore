# Демо-проект для защиты ВКР

Документ описывает тестовый проект, на котором можно быстро показать основные преимущества платформы: управление требованиями, workflow задачи, LangGraph-валидацию, RAG-память, чат-агентов, предложения изменений, мониторинг, Qdrant и eval-контур.

Демонстрационная предметная область: **Цифровая платформа заявок на услуги университета**.

## 1. Назначение сценария

Цель сценария - подготовить небольшой, понятный комиссии проект, в котором одна и та же бизнес-область проходит через все ключевые механизмы системы:

- требование создается и проходит проверку качества;
- неполное требование получает замечания и уточняющие вопросы;
- вложения и результаты валидации попадают в RAG-память;
- QA Agent отвечает с учетом контекста задачи и похожих задач проекта;
- ChangeTracker Agent выделяет предложение изменения из коммуникации;
- администратор видит трассировку графов, LLM-запросы, Qdrant coverage и eval-результаты.

Оптимальная длительность показа: 7-10 минут. Расширенный вариант с eval-запусками и мониторингом занимает 15-20 минут.

## 2. Что показывает демонстрация

Демонстрация закрывает четыре тезиса ВКР.

| Тезис | Что показать в системе |
| --- | --- |
| Требование является управляемым объектом | Проект, задача, роли, статусы, подтверждение, разработка, тестирование |
| Качество постановки можно проверять автоматически | Запуск валидации, verdict `approved` или `needs_rework`, issues и questions |
| Коммуникация превращается в артефакты | Чат задачи, QA-ответы, предложения изменений, вопросы валидации |
| Проектный контекст переиспользуется | RAG/Qdrant, cross-task ответы, eval-метрики, graph runs |

## 3. Подготовка окружения

Из корня репозитория подготовьте backend-конфиг:

```powershell
if (-not (Test-Path backend\.env)) { Copy-Item backend\.env.example backend\.env }
```

В `backend/.env` задайте минимум:

```env
JWT_SECRET_KEY=replace-with-long-random-secret
LLM_SETTINGS_ENCRYPTION_KEY=replace-with-another-long-random-secret
EMBEDDING_PROVIDER=ollama
OLLAMA_EMBEDDING_MODEL=nomic-embed-text
```

Если для демонстрации используется облачный LLM-провайдер, дополнительно задайте его ключи или настройте профиль через UI в разделе `Администрирование -> Модельные профили`.

Запустите контур:

```powershell
docker compose up --build -d
```

Проверьте доступность:

- UI: `http://localhost:8080`
- frontend health: `http://localhost:8080/healthz`
- backend health: `http://localhost:8080/api/healthz`
- backend readiness: `http://localhost:8080/api/readyz`

Если LLM-провайдер еще не настроен, рабочий CRUD-контур и роли можно показать сразу, а AI-сценарии лучше демонстрировать после проверки профиля в админке.

## 5. Создание проекта

Создайте проект:

- название: `Цифровая платформа заявок на услуги университета`;
- описание: `Система для подачи, согласования и сопровождения заявок студентов и сотрудников на услуги университета: справки, доступы, аудитории, оборудование и консультации.`;

Добавьте участников проекта:

- Анна Аналитик - аналитик требований;
- Дмитрий Разработчик - исполнитель разработки;
- Татьяна Тестировщик - проверка результата;
- Марина Менеджер - контроль сроков и приоритетов.

| --- | --- | --- |

## 7. Демонстрационные задачи

Создайте задачи в указанном порядке. Это даст материал для RAG и cross-task вопросов.

### 7.1 Хорошее требование: создание заявки на справку

Название: `Подача заявки на получение справки об обучении`

Теги: `requests`, `notifications`, `sla`

Описание:

```md
## Контекст
Студент должен иметь возможность подать заявку на получение справки об обучении через личный кабинет.

## Пользователи
- Студент создает заявку и отслеживает статус.
- Сотрудник деканата рассматривает заявку.
- Менеджер учебного офиса контролирует просроченные заявки.

## Сценарий
1. Студент открывает раздел "Услуги университета".
2. Выбирает услугу "Справка об обучении".
3. Заполняет цель получения справки и язык документа.
4. Отправляет заявку.
5. Система присваивает номер заявки и статус "Новая".

## Бизнес-правила
- Заявка должна быть обработана не позднее 3 рабочих дней.
- При смене статуса студент получает уведомление в личном кабинете.
- Сотрудник деканата видит только заявки своего факультета.

## Критерии приемки
- После отправки заявки пользователь видит номер и текущий статус.
- Сотрудник деканата может принять заявку в работу.
- При изменении статуса создается уведомление для студента.
- Просроченная заявка отображается менеджеру учебного офиса.
```

Ожидаемый результат: после валидации задача получает `approved` и переходит к статусу `Ожидает подтверждения`.

### 7.2 Неполное требование: бронирование аудитории

Название: `Бронирование аудитории для мероприятия`

Теги: `requests`, `approval`

Описание:

```md
Пользователь должен иметь возможность забронировать аудиторию для мероприятия. Система должна показать доступные аудитории и отправить заявку ответственному сотруднику.
```

Ожидаемый результат: валидация возвращает `needs_rework`, потому что не указаны роли, критерии приемки, сроки согласования, ограничения по вместимости и правила уведомлений.

Что сказать на защите: "Здесь видно, что система не просто хранит карточку, а помогает аналитику улучшить постановку до передачи в разработку."

### 7.3 Требование с вложением: заявка на доступ к лаборатории

Название: `Заявка на доступ к компьютерной лаборатории`

Теги: `auth`, `requests`, `attachments`, `audit`

Описание:

```md
Сотрудник кафедры должен подать заявку на доступ студенческой группы к компьютерной лаборатории. В заявке указываются группа, период доступа, преподаватель, список студентов и основание для доступа.

Критерии приемки:
- Пользователь может приложить список студентов файлом.
- Ответственный сотрудник видит вложение в карточке заявки.
- После одобрения доступ фиксируется в журнале аудита.
- Отклонение заявки требует указания причины.
```

Создайте локальный текстовый файл `lab-access-policy.txt` и загрузите его как вложение:

```text
Доступ в компьютерную лабораторию выдается только студентам из утвержденного списка.
Минимальный срок рассмотрения заявки - 1 рабочий день.
Максимальный срок действия доступа - до конца текущего семестра.
Все решения по доступу должны попадать в журнал аудита с указанием ответственного сотрудника.
```

Ожидаемый результат: после индексации QA Agent сможет ссылаться на правила из вложения.

### 7.4 Задача для cross-task RAG: уведомления по заявкам

Название: `Уведомления о смене статуса заявки`

Теги: `notifications`, `requests`

Описание:

```md
Система должна отправлять пользователю уведомления при изменении статуса заявки. Уведомление отображается в личном кабинете и содержит номер заявки, новый статус, дату изменения и краткий комментарий ответственного сотрудника.

Критерии приемки:
- При переводе заявки в статус "В работе" создается уведомление.
- При переводе заявки в статус "Отклонена" уведомление содержит причину отказа.
- При завершении заявки пользователь получает сообщение о готовности результата.
- История уведомлений доступна в карточке заявки.
```

Ожидаемый результат: эта задача станет источником проектного контекста для вопросов из других задач.

### 7.5 Задача с предложением изменения: согласование заявки

Название: `Согласование заявки на выдачу оборудования`

Теги: `approval`, `requests`, `audit`

Описание:

```md
Сотрудник университета должен иметь возможность подать заявку на выдачу оборудования для мероприятия. Ответственный сотрудник рассматривает заявку и принимает решение.

Критерии приемки:
- Пользователь создает заявку с описанием мероприятия и перечнем оборудования.
- Ответственный сотрудник принимает или отклоняет заявку.
- Решение сохраняется в истории заявки.
```

Ожидаемый результат: задача подходит для показа ChangeTracker Agent, потому что в нее удобно внести уточнение через чат.

## 8. Сценарий показа: рабочий контур

1. Войдите под аналитиком или администратором.
2. Откройте проект `Цифровая платформа заявок на услуги университета`.
3. Покажите список задач и фильтрацию по статусам.
4. Откройте задачу `Подача заявки на получение справки об обучении`.
5. Запустите валидацию.
6. Покажите результат проверки, issues/questions и статус `Ожидает подтверждения`.

Что сказать на защите: "Система связывает работу аналитика, разработчика и тестировщика. Требование проходит проверку качества до разработки, а дальнейший workflow фиксирует ответственность и состояние работы."

## 9. Сценарий показа: AI/RAG-контур

### 9.1 Валидация неполного требования

Откройте задачу `Бронирование аудитории для мероприятия` и запустите валидацию.

Покажите:

- verdict `needs_rework`;
- список замечаний;
- уточняющие вопросы;
- связь замечаний с правилами проекта.

Фраза для защиты: "Это демонстрирует раннее выявление дефектов требований: неполная постановка не передается дальше по процессу без доработки."

### 9.2 QA Agent по текущей задаче и вложению

Откройте чат задачи `Заявка на доступ к компьютерной лаборатории`.

Отправьте сообщение:

```text
@qa Какие ограничения по сроку действия доступа нужно учитывать?
```

Ожидаемый ответ: агент должен использовать контекст задачи и вложение `lab-access-policy.txt`, упомянуть ограничение "до конца текущего семестра" и срок рассмотрения.

Что сказать на защите: "Вложения не остаются просто файлами. Их содержимое превращается в индексируемый контекст, который затем использует агент."

### 9.3 Cross-task RAG

В чате задачи `Подача заявки на получение справки об обучении` отправьте:

```text
@qa Какие уведомления должен получить пользователь при изменении статуса заявки?
```

Ожидаемый ответ: агент должен использовать контекст задачи `Уведомления о смене статуса заявки` и перечислить уведомления при переходе в работу, отклонении и завершении.

Что сказать на защите: "Это показывает проектную память: агент использует не только текущую карточку, но и релевантные знания из других задач проекта."

### 9.4 ChangeTracker Agent

Откройте чат задачи `Согласование заявки на выдачу оборудования`.

Отправьте сообщение:

```text
@change-tracker Нужно добавить правило: если стоимость оборудования выше 100 000 рублей, заявка должна дополнительно согласовываться менеджером учебного офиса.
```

Ожидаемый результат:

- появится agent message;
- будет создано предложение изменения со статусом `Новое`;
- после принятия предложения требование можно обновить и повторно провалидировать.

Что сказать на защите: "Коммуникация не теряется в переписке. Из сообщения выделяется структурированный артефакт - предложение изменения, которое можно принять или отклонить."

## 10. Сценарий показа: административный и eval-контур

Откройте `Администрирование`.

### 10.1 Модельные профили

Раздел: `Модельные профили`.

Покажите:

- provider kind;
- модель;
- base URL;
- тест профиля;
- стоимость токенов, если она настроена.

Фраза для защиты: "LLM-провайдеры управляются через runtime-слой. Бизнес-логика графов не привязана к одному конкретному провайдеру."

### 10.2 Промпты агентов

Раздел: `Промпты агентов`.

Покажите agent keys:

- `task-validation`;
- `chat-routing`;
- `qa-answer`;
- `qa-verifier`;
- `change-tracker`.

Фраза для защиты: "Промпты вынесены в администрируемый слой, поэтому поведение агентов можно уточнять без изменения кода."

### 10.3 Qdrant и RAG

Раздел: `Qdrant и RAG`.

Покажите:

- overview коллекций;
- coverage задач;
- probe по вопросу из демо;
- ручной resync задачи, если нужно.

Пример probe:

```text
Какие уведомления получает пользователь при смене статуса заявки?
```

Фраза для защиты: "Qdrant используется как проектная память. Здесь видно, какие задачи проиндексированы и какие фрагменты возвращаются по контрольному запросу."

### 10.4 Графы

Раздел: `Графы`.

Покажите последний запуск:

- `validation_graph`;
- `chat_graph`;
- `rag_retrieval_graph`;
- `qa_agent_graph`;
- `change_tracker_agent_graph`.

Фраза для защиты: "LangGraph дает трассируемость AI-сценариев: можно видеть узлы, переходы, статусы и ошибки выполнения."

### 10.5 LLM-запросы

Раздел: `LLM-запросы`.

Покажите:

- agent key;
- provider/model;
- latency;
- prompt tokens;
- completion tokens;
- total tokens;
- estimated cost, если настроена стоимость.

Фраза для защиты: "Для ВКР важно, что AI-контур не является черным ящиком: вызовы логируются, измеряются и могут быть проанализированы."

### 10.6 Eval-страницы

Разделы:

- `Оркестратор`;
- `RAG Eval`;
- `Adaptation Eval`;
- `Validation Eval`;
- `QuRE Eval`.

Минимально покажите `RAG Eval`:

1. импорт датасета или выбор существующего;
2. запуск с `Query rewriter`, `Hybrid rerank`, `Cross-task context` и `BM25 baseline`;
3. summary metrics: `recall_at_1`, `recall_at_3`, `recall_at_5`, `mrr`, `bm25_mrr`, `rag_vs_bm25_mrr_delta`;
4. CSV/JSON export результатов.

Фраза для защиты: "Экспериментальная часть ВКР строится не на субъективной демонстрации, а на eval-наборах и метриках."

## 11. Быстрый сценарий на 7-10 минут

Используйте этот маршрут, если время ограничено.

1. `Проекты`: открыть демо-проект и список задач.
2. `Подача заявки на получение справки об обучении`: показать хорошо оформленное требование и успешную валидацию.
3. `Бронирование аудитории для мероприятия`: запустить валидацию и показать `needs_rework`.
4. `Заявка на доступ к компьютерной лаборатории`: задать вопрос `@qa` по вложению.
5. `Согласование заявки на выдачу оборудования`: отправить сообщение `@change-tracker` и показать предложение изменения.
6. `Администрирование -> Графы`: показать трассировку последнего запуска.
7. `Администрирование -> Qdrant и RAG`: показать coverage и probe.

Короткая итоговая фраза: "Система усиливает процесс управления требованиями за счет автоматической проверки качества, сохранения коммуникации в виде артефактов и переиспользования проектного контекста через RAG."

## 12. Расширенный сценарий на 15-20 минут

1. Показать создание проекта, участников и кастомных правил.
2. Создать или открыть все пять демонстрационных задач.
3. Провести полный workflow для хорошего требования.
4. Провести валидацию неполного требования.
5. Показать QA Agent по вложению.
6. Показать cross-task RAG.
7. Показать ChangeTracker Agent и рассмотрение предложения изменения.
8. Открыть `Графы` и показать трассировку `validation_graph` и `chat_graph`.
9. Открыть `LLM-запросы` и показать latency/tokens/cost.
10. Открыть `RAG Eval` и показать сравнение с BM25 baseline.
11. Открыть `Adaptation Eval` и объяснить, как коммуникация превращается в вопросы и контекст для будущей валидации.
12. Завершить демонстрацию связью с гипотезой ВКР.

Финальная фраза для защиты: "Ключевой результат работы - не отдельный чат-бот, а платформа, в которой требования, коммуникация, проверки, предложения изменений и проектная память объединены в управляемый процесс."

## 13. Ожидаемые результаты

После прохождения сценария в системе должны быть видны:

- проект с участниками разных ролей;
- задачи со статусами `Черновик`, `Нужна доработка`, `Ожидает подтверждения`, `Готово к разработке`, `Разработка`, `Готово к тестированию`, `Тестирование`, `Задача выполнена`;
- минимум одна успешная валидация с verdict `approved`;
- минимум одна неуспешная валидация с verdict `needs_rework`;
- QA-ответ по вложению;
- QA-ответ с использованием cross-task RAG;
- предложение изменения со статусом `Новое`, `Принято` или `Отклонено`;
- записи в `Графы`;
- записи в `LLM-запросы`;
- coverage в `Qdrant и RAG`;
- при расширенном сценарии - eval-запуск с summary metrics.

## 14. Troubleshooting

### Не работает LLM provider

Что проверить:

- в `backend/.env` заданы ключи и base URL;
- в `Администрирование -> Модельные профили` профиль активен;
- тест профиля проходит успешно;
- выбранная модель поддерживает нужный режим: обычный text, embeddings или Vision;
- контейнер backend был перезапущен после изменения `.env`.

Быстрое действие:

```powershell
docker compose restart backend
```

Если облачный провайдер недоступен, покажите CRUD/workflow, Qdrant coverage и сохраненные ранее eval-результаты, а AI-вызовы объясните по трассировке существующих запусков.

### Qdrant пустой или не возвращает контекст

Что проверить:

- контейнер `qdrant` запущен;
- backend readiness возвращает `200`;
- задача была создана или обновлена после запуска Qdrant;
- в `Администрирование -> Qdrant и RAG` есть coverage по проекту;
- у задачи есть текст или поддерживаемое текстовое вложение;
- embeddings provider настроен корректно.

Быстрое действие:

1. Откройте `Администрирование -> Qdrant и RAG`.
2. Найдите задачу демо-проекта.
3. Запустите ручной `resync`.
4. Повторите probe.

### WebSocket не обновляет чат

Что проверить:

- пользователь авторизован;
- access token не истек;
- backend доступен через `/api`;
- страница чата открыта по маршруту задачи;
- reverse proxy или браузер не блокирует WebSocket.

Быстрое действие: обновите страницу, выйдите и войдите заново, затем повторите отправку сообщения.

### Eval-запуск завершился ошибкой

Что проверить:

- выбран корректный dataset;
- в dataset есть project id и task external ids;
- LLM-провайдер для judge доступен;
- embeddings provider доступен;
- Qdrant содержит коллекции;
- в конфигурации eval не выбрано больше 3 judge provider profiles.

Быстрый вариант для защиты: открыть последний успешный запуск и показать summary metrics, case results и export. Для ВКР достаточно показать воспроизводимость контура и структуру метрик, даже если новый live-запуск в момент защиты недоступен.
