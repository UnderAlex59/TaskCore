# Ручной eval-набор для проверки качества RAG

Файл предназначен для ручного заведения данных в тестовую зону. Сначала создайте задачи из раздела "Текстовки задач", затем задавайте вопросы из раздела "Eval-вопросы" и фиксируйте результат по метрикам ниже.

## Метрики

Оценивайте каждый прогон по шкале `0..2`, где `0` - плохо, `1` - частично, `2` - хорошо.

| Метрика | Что проверять |
| --- | --- |
| `source_found` | Найден правильный источник: нужная задача, вложение или фрагмент текста. |
| `no_hallucination` | Ответ не содержит фактов, которых нет в найденных документах. |
| `answer_completeness` | Ответ достаточно полный для вопроса пользователя. |
| `groundedness` | Ключевые утверждения явно следуют из найденных документов. |
| `no_external_context` | Модель не использует внешний или выдуманный контекст. |

Рекомендуемый формат фиксации результата:

```text
ID:
Ответ модели:
Найденные источники:
source_found:
no_hallucination:
answer_completeness:
groundedness:
no_external_context:
Итоговый балл:
Заметки:
```

## Текстовки задач

### RAG-TASK-001. Статусы рабочей документации

**Теги:** `rd`, `status`, `workflow`

**Описание задачи для тестовой зоны:**

Нужно реализовать справочник статусов рабочей документации. В справочнике должны быть статусы: `Черновик`, `На проверке`, `Возвращено на доработку`, `Согласовано`, `Архив`. Статус `Черновик` присваивается новой рабочей документации. Статус `На проверке` ставится после отправки документа ответственному за проверку. Статус `Возвращено на доработку` используется, если проверяющий оставил замечания. Статус `Согласовано` означает, что проверка завершена без блокирующих замечаний. Статус `Архив` используется для документов, которые больше не участвуют в текущей работе.

### RAG-TASK-002. Рассылка по статусам рабочей документации

**Теги:** `rd`, `notifications`, `status`

**Описание задачи для тестовой зоны:**

Нужно реализовать рассылку уведомлений по изменениям статусов рабочей документации. При переходе в статус `На проверке` уведомление получает ответственный за проверку. При переходе в статус `Возвращено на доработку` уведомление получает автор документа. При переходе в статус `Согласовано` уведомление получают автор документа и руководитель проекта. Рассылка должна содержать номер документа, новый статус, дату изменения и ссылку на карточку документа. Расписание cron, SMTP-настройки и текст email-шаблонов в этой задаче не задаются.

### RAG-TASK-003. Замечания к рабочей документации

**Теги:** `rd`, `review`, `comments`

**Описание задачи для тестовой зоны:**

Нужно реализовать учет замечаний к рабочей документации. У замечания должны храниться: идентификатор документа, автор замечания, текст замечания, дата создания, статус замечания и ссылка на раздел документа. Доступные статусы замечания: `Открыто`, `Исправлено`, `Отклонено`. Замечания создаются только в статусе документа `На проверке`. После исправления автор документа переводит замечание в статус `Исправлено`, а проверяющий подтверждает результат или возвращает замечание в `Открыто`.

### RAG-TASK-004. Ответственные за проверку рабочей документации

**Теги:** `rd`, `responsible`, `review`

**Описание задачи для тестовой зоны:**

Нужно добавить назначение ответственных за проверку рабочей документации. Для каждого документа должен быть один основной проверяющий и до трех дополнительных проверяющих. Основной проверяющий принимает финальное решение о согласовании. Дополнительные проверяющие могут оставлять замечания, но не переводят документ в статус `Согласовано`. Если основной проверяющий не назначен, документ нельзя отправить в статус `На проверке`.

### RAG-TASK-005. История статусов документов

**Теги:** `rd`, `history`, `audit`

**Описание задачи для тестовой зоны:**

Нужно реализовать историю изменений статусов документов рабочей документации. Для каждой смены статуса нужно хранить: идентификатор документа, старый статус, новый статус, пользователя, дату изменения и комментарий. Историю можно выгружать за период по одному документу или по проекту. Формат выгрузки в задаче не закреплен. История должна показывать только изменения статусов, а не изменения текста документа.

### RAG-TASK-006. Категории статусов рабочей документации

**Теги:** `rd`, `status`, `category`

**Описание задачи для тестовой зоны:**

Нужно добавить категории статусов рабочей документации для аналитики. Категория `В работе` включает статусы `Черновик` и `Возвращено на доработку`. Категория `Проверка` включает статус `На проверке`. Категория `Завершено` включает статусы `Согласовано` и `Архив`. Категории не заменяют сами статусы и не должны отображаться пользователю как доступные статусы документа.

### RAG-TASK-007. Расширенный импорт реестра рабочей документации

**Теги:** `rd`, `import`, `registry`

**Описание задачи для тестовой зоны:**

Нужно реализовать расширенный импорт реестра рабочей документации. Импортируемые поля: номер документа, название документа, текущий статус, автор, дата создания, дата последнего изменения и проект. Если документ с таким номером уже существует в проекте, нужно обновить его текущий статус и дату последнего изменения. Если документ не найден, нужно создать новую карточку. Импорт не должен создавать историю статусов задним числом.

### RAG-TASK-008. Атрибут шага прогресса

**Теги:** `progress`, `steps`, `attribute`

**Описание задачи для тестовой зоны:**

Нужно добавить атрибут `Шаг прогресса` для задач проекта. Атрибут выбирается из справочника шагов прогресса и используется для группировки задач на доске прогресса. У одной задачи может быть только один текущий шаг прогресса. Изменение шага прогресса не меняет статус задачи и не влияет на статусы рабочей документации.

### RAG-TASK-009. Справочник групп шагов прогресса

**Теги:** `progress`, `steps`, `dictionary`

**Описание задачи для тестовой зоны:**

Нужно создать справочник групп шагов прогресса. Группа объединяет несколько шагов прогресса для отчетности. У группы должны быть название, код, сортировка и признак активности. Неактивные группы не предлагаются при настройке новых шагов, но сохраняются в исторических отчетах. Группы шагов прогресса не являются категориями статусов рабочей документации.

### RAG-TASK-010. Экспорт замечаний по рабочей документации

**Теги:** `rd`, `export`, `comments`

**Описание задачи для тестовой зоны:**

Нужно реализовать экспорт замечаний по рабочей документации. Экспорт должен включать номер документа, автора замечания, текст замечания, статус замечания, дату создания и ссылку на раздел документа. Экспорт выполняется по проекту и периоду. Экспорт не должен включать историю статусов документа, если пользователь явно не выбрал отдельный отчет по истории.
